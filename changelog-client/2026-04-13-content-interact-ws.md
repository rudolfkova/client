# Клиент: каталог предметов и сообщение `interact` (game-service)

Дата: 2026-04-13. Описано только **новое** относительно уже существующего WS-протокола (`service: "game"`, `move`, `hit`, `spawn_tile`, `clear_tile`, `state`, …).

---

## 1. Новый тип входящего сообщения (клиент → сервер)

**`type`:** `"interact"` (константа в Go: `gamekit.TypeInteract`).

**`payload`** (JSON), структура как у `gamekit.InteractIntent`:

| Поле           | Тип    | Обязательно | Смысл |
|----------------|--------|---------------|--------|
| `item_def_id`  | string | да (непустая после trim) | Ключ предмета из **каталога** на сервере (`catalog.json`), у которого задан блок `interact`. |

Пример конверта:

```json
{
  "service": "game",
  "type": "interact",
  "payload": {
    "item_def_id": "lever_sample"
  }
}
```

Сервер **не** шлёт `reject` при ошибочном/неизвестном `interact`: при невалидном JSON payload, пустом `item_def_id`, неизвестном id или у предмета нет `interact` — действие **молча игнорируется** (как у `move`/`spawn_tile` с битым payload). Клиенту для отладки полезно логировать исходящий запрос; подтверждения кроме очередного broadcast **`state`** нет.

---

## 2. Когда `interact` вообще обрабатывается

На **game-service** в конфиге должны быть заданы пути к данным контента (в Docker по умолчанию это смонтированные файлы под `/app/data/content/`):

- каталог предметов: JSON **catalog**;
- каталог файлов **сценариев** (отдельные JSON-файлы шагов).

Если каталог **не** загружен (пути пустые или ошибка при старте сервера), сообщения `interact` не имеют эффекта.

Клиенту достаточно знать: **фича активна только если сервер поднят с контентом**; иначе не ждать смены мира от `interact`.

---

## 3. Модель данных на сервере (для синхронизации с редактором / клиентом)

Форматы лежат в репозитории в `game-service/data/content/` (пример) и дублируют контракт пакета `pkg/gamekit/content` (тот же JSON можно парсить в Go на стороне редактора).

### 3.1. `catalog.json`

Корень:

- `schema_version` (number), сейчас **1**;
- `items` — объект, ключи — строковые id (часто совпадают с `id` внутри записи).

Каждый предмет:

| Поле        | Тип    | Смысл |
|-------------|--------|--------|
| `id`        | string | Идентификатор (должен совпадать с ключом в `items`, если ключ используется как `item_def_id`). |
| `name`      | string | Отображаемое имя (сервер в этом PR не шлёт его в `state`; для редактора/тулзов). |
| `pickable`  | bool   | Зарезервировано под будущий подбор в инвентарь; **поведение подбора пока не реализовано**. |
| `interact`  | object \| отсутствует | Если есть — у предмета есть сценарий по нажатию «взаимодействовать». |

Блок **`interact`**:

| Поле     | Тип    | Смысл |
|----------|--------|--------|
| `script` | string | Имя файла сценария в каталоге скриптов, например `"lever_sample.json"`. |
| `args`   | object | Произвольные JSON-значения; **мержатся в args каждого шага** сценария (шаг перекрывает каталог при совпадении ключей). |

### 3.2. Файл сценария (например `scripts/lever_sample.json`)

| Поле             | Тип    | Смысл |
|------------------|--------|--------|
| `schema_version` | number | Сейчас **1**. |
| `steps`          | array  | Последовательность шагов. |

Каждый элемент **`steps`**:

| Поле  | Тип    | Смысл |
|-------|--------|--------|
| `op`  | string | Имя операции, известное серверу. |
| `args`| object | Опционально; мерж с `interact.args` из каталога (приоритет у полей шага). |

---

## 4. Какие `op` сейчас есть (чтобы клиент не ожидал лишнего в ответе)

Сервер исполняет шаги **синхронно в том же тике**, что и действие `interact`. Исходящих отдельных событий «сценарий выполнен» **нет** — эффекты видны в следующем **`state`** (например, новые тайлы).

Встроенные op (пакет `content`, всегда есть):

| `op`        | `args` | Эффект |
|-------------|--------|--------|
| `noop`      | —      | Ничего. |
| `debug_log` | `message` (string) | Пишет строку в лог сервера (уровень debug); на клиент не уходит. |

Игровой op (регистрируется движком):

| `op`               | Основные `args` | Эффект |
|--------------------|-----------------|--------|
| `world_spawn_tile` | `x`, `y` (number), `texture` (string), опционально `layer`, `rotation` (number, по умолчанию 0), `blocks` (bool) | Как ручной `spawn_tile`: в клетке на слое заменяется тайл на указанный. |

Любой **неизвестный** `op` в сценарии: сервер логирует предупреждение, шаг падает с ошибкой; остальное поведение — по текущей реализации раннера (клиенту не гарантируется частичное применение; для UX безопаснее считать сценарий атомарным только после стабилизации контракта).

---

## 5. Что клиенту нужно реализовать

1. Отправка конверта с **`type: "interact"`** и **`payload.item_def_id`**, совпадающим с id из серверного каталога (клиент может держать копию `catalog.json` или получать список id иначе — **сервер id с клиента не отдаёт в `state` в этом PR**).
2. UI: выбор цели взаимодействия (рычаг, объект) → подстановка правильного `item_def_id`.
3. Не ожидать отдельного ack: после действия приходит обычный **`state`**; если контент отключён или id неверный — состояние может не измениться без ошибки в WS.

## Кратко одной строкой для промпта

Добавлен WS **`interact`** с `payload: { "item_def_id": "<string>" }`; сервер по каталогу и JSON-сценариям выполняет шаги (`noop`, `debug_log`, `world_spawn_tile`); ответа кроме обычного **`state`** нет; при ошибках/отключённом контенте — тихий no-op.
